# 2026-W15 (Apr 6–12) — Demo site ships, Constellation gets polished

**Headline**: Omniscitus went from "plugin you install" to "plugin you can
actually *see* before installing". The demo site landed, the Constellation
view got the performance and polish it needed, and the repo started
dogfooding itself.

## What happened

- **Demo birdview on the docs site** (#29). Static snapshot of birdview
  HTML + data, served under `/birdview-demo/` on the omniscitus landing.
  WebGL graceful fallback for browsers that can't render Three.js.
- **Dogfooding** (#28). Seeded this repo's own `.omniscitus/` so the
  demo has meaningful content to show instead of an empty state.
- **Demo path fixes** (#33, #34). Source birdview pages hardcode
  `/blueprint`, `/history` etc. because they're served from the plugin's
  Node server. Rewritten to relative paths for Vercel hosting. Snapshotted
  `/api/prompt-tests` with a graceful 404 fallback.
- **Constellation polish** (#25, #26, #27). Folder-level selection, tree
  performance via event delegation and debouncing, card size balanced
  against the other Birdview pills. Constellation moved to the bottom of
  the landing as a visual teaser rather than a lead feature.
- **Install UX** (#31, #32). Install instructions switched to
  slash-command format. The terminal layout stopped laying comments
  next to commands on narrow viewports.
- **Unit test infrastructure** (#30). 42 tests covering birdview parsers,
  seed script, demo build. Safety net for all the parser work that
  came in W16.
- **v0.2.0** (#35). Hooks fix + constellation dim effect + version bump.

## Decisions worth remembering

- Constellation is the visual hook, not the primary UI. It lives at the
  bottom of the landing for a reason — you come for Blueprint / History /
  Tests and stay for Constellation.
- Demo site uses static snapshots, not a live Node server. Everything is
  pre-rendered and JSON endpoints are files under `data/`. Much cheaper
  to host, survives Vercel cold starts.
