# 0.9 Public Release Gate — execution debt

**Participants**: claude, dan
**Status**: open
**Created**: 2026-04-14
**Last updated**: 2026-04-14

## Intent

The thesis is 10/10 — "records compound, tools don't", records are the
real invariant of AI-driven production. The current execution is ~7/10.
Before omniscitus goes public as a stable 0.9, the gap between
"defensible thesis" and "defensible implementation" needs to close.

This unit tracks the specific debts. When they are all done, we ship
0.9 as the public-launch version.

## Context

- **Background**: 0.2 → 0.8 in two days shipped a lot of surface area
  but quality is uneven. 14 patch releases in one day exposed real
  gaps — banner saga, `_..yaml` leak regression, tree-panel max-height,
  hint-bar collision, Participants that are backfill-only, tests as
  catalog without runner, no history search.
- **Constraints**: keep the thesis clean — records, not features.
  Don't add new surfaces before stabilizing existing ones.

## Pending

### Must-fix before 0.9 (data integrity)

- [ ] **History search** — grep across unit bodies. Currently
  date/domain/participant filters exist, but you can't find "that
  CEFR decision from a month ago" without eyeballing. This is the
  #1 retrieval gap for anyone actually using the tool.
- [ ] **Participants real-data path**. Right now Participants on every
  unit means "whatever migrate backfilled from member folders" + "claude".
  /wrap-up should detect actual session participants from `git log %an`
  on touched files within the session window, not rely on manual entry
  or backfill heuristics.
- [ ] **Scale test on 10k+ files**. Birdview loads the full blueprint
  tree client-side. Verify behavior on a repo with 10,000+ tracked files:
  initial load time, filter responsiveness, Constellation node cap
  (currently hard-cap 2,000 with truncation banner — decide if that
  limit is acceptable or needs pagination).
- [ ] **Test runner integration**. meta.yaml indexes real tests but
  doesn't run them. At minimum: a shell-out to the declared `framework:`
  (vitest/jest/pytest/node-test), parsed pass/fail summary shown on
  the Test card. Without this, the "Tests" surface is a catalog, not
  a test dashboard.
- [ ] **Living memory reconciliation**. Units are append-only
  snapshots. If a decision reverses a week later, old unit still
  shows old decision. Design a `superseded_by:` / `reconciled_with:`
  pointer so units can form a chain, and birdview can show the latest
  verdict.

### Should-fix before 0.9 (trust and polish)

- [ ] **Review system**. Currently "Reviews for this page" is a dead
  feature — no docs, no onboarding. Either: (a) wire it into wrap-up so
  reviews become part of the unit trail, or (b) remove until there's a
  real use case.
- [ ] **Stop same-day patch releases**. Minimum 24-hour bake between
  version bump and next one. If a user-facing bug comes in mid-day,
  queue it on a `next-patch` branch and ship as a batch. Prevents the
  banner-saga pattern.
- [ ] **Regression tests for hooks + UI rendering**. 125 unit tests
  cover parsers but not the Blueprint/History/Tests HTML render paths.
  Playwright (or equivalent) smoke test: load each page, screenshot,
  assert presence of key elements. Prevents "banner invisible on
  constellation" class of regressions.
- [ ] **"Live demo" is actually a static snapshot**. The banner says
  "live demo of birdview running against the omniscitus repo". Either
  change the wording to "snapshot demo" for honesty, or invest in a
  read-only live backend (Vercel function + Blob snapshot of .omniscitus/).

### Nice-to-have (brand + docs)

- [ ] **Landing + README reframe around "records compound" thesis**
  (tracked separately — feat/landing-thesis-reframe branch).
- [ ] **Session-end nudge**. Detect via SessionEnd or PreCompact hook
  when a session has meaningful touches and prompt `/wrap-up`. Habit
  formation — right now wrap-up relies on user remembering.
- [ ] **Migration for non-Claude-Code editors**. Document what
  omniscitus does and doesn't offer if you're using Cursor/Copilot.
  Either: minimal value (blueprints can be populated via `/blueprint-sync`
  only) or: explicit "Claude Code required" label.

## Exit criteria for 0.9 release

All **Must-fix** items closed. At least 3 of the 4 **Should-fix** items
closed. Nice-to-haves can slide to 0.9.x.

On ship-day, post one honest X/Threads update:
"0.9 — the version we would have launched if we'd skipped the hype."

## Notes

This list is a direct response to a cold evaluation by Claude in the
2026-04-14 session. The evaluation was harsh but right on most points.
It's worth preserving the original critique because the same issues
will resurface in future plugin work:

1. Features must be earned by pain, not designed as a parade.
2. Records are the invariant. Tooling churns.
3. Fake data (seeded Participants, catalog-only Tests) erodes trust
   faster than missing data does.
4. Same-day patch velocity is a red flag, not a badge.
