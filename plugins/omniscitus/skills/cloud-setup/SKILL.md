# Cloud Setup — Transition to Real-Time Sync

Generate the configuration scaffold and architecture guide for organizations
that want to move from git-based `.omniscitus/` storage to a cloud-backed
real-time sync system.

## When to Use

- User types `/cloud-setup`
- User says "클라우드 설정", "omniscitus cloud", "real-time sync"
- Organization needs concurrent editing without git merge conflicts
- Team wants a shared dashboard accessible without localhost

## Instructions

### Step 1: Assess Current State

```bash
# Count team members (from blueprint source labels)
grep -r "source: " .omniscitus/blueprints/ 2>/dev/null | sed 's/.*source: //' | sort -u

# Count total tracked files
grep -r "status: active" .omniscitus/blueprints/ 2>/dev/null | wc -l

# Count history units
cat .omniscitus/history/_index.yaml 2>/dev/null | grep "id:" | wc -l
```

### Step 2: Ask Requirements

Use AskUserQuestion:
- "Which cloud provider do you prefer? (AWS, GCP, Azure, or self-hosted)"
- "How many concurrent developers will use omniscitus?"
- "Do you need the shared birdview dashboard accessible from a URL?"

### Step 3: Generate Cloud Config

Create `.omniscitus/cloud.yaml`:

```yaml
# Omniscitus Cloud Configuration
# Generated by /cloud-setup
# This file configures the cloud sync layer for real-time collaboration.

version: 1
enabled: false  # Set to true when cloud backend is deployed

# Cloud backend connection
backend:
  # Options: "s3", "gcs", "azure-blob", "supabase", "custom"
  provider: "{user_choice}"
  
  # Provider-specific settings (fill in after deployment)
  config:
    # S3 example:
    # bucket: "omniscitus-{project-name}"
    # region: "us-east-1"
    # prefix: ".omniscitus/"
    
    # Supabase example (recommended for small teams):
    # url: "https://xxx.supabase.co"
    # anon_key: "your-anon-key"
    # table_prefix: "omniscitus_"

# Sync behavior
sync:
  # How often to push local changes to cloud (seconds)
  push_interval: 30
  
  # How often to pull remote changes (seconds)
  pull_interval: 15
  
  # Conflict resolution strategy
  # "last-write-wins" — simple, may lose data
  # "merge" — attempts to merge changes (recommended)
  # "manual" — flags conflicts for manual resolution
  conflict_strategy: "merge"
  
  # Which data to sync (all by default)
  include:
    - blueprints
    - history
    - ontology
    - tests
  
  # Exclude patterns
  exclude: []

# Shared dashboard
dashboard:
  # Set to true to deploy birdview as a shared web app
  enabled: false
  # URL will be set after deployment
  url: ""

# Notifications (optional)
notifications:
  # Notify when a team member runs /wrap-up
  on_wrap_up: false
  # Notify on blueprint conflicts
  on_conflict: true
  # Webhook URL (Slack, Discord, etc.)
  webhook_url: ""
```

### Step 4: Generate Architecture Guide

Create `.omniscitus/cloud-architecture.md`:

```markdown
# Omniscitus Cloud Architecture Guide

## Overview

This guide helps you deploy a cloud backend for omniscitus real-time sync.
The cloud layer sits between local `.omniscitus/` files and a central store,
keeping all team members in sync without relying solely on git.

## Architecture

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Developer A │     │  Developer B │     │  Developer C │
│  .omniscitus/│     │  .omniscitus/│     │  .omniscitus/│
└──────┬───────┘     └──────┬───────┘     └──────┬───────┘
       │                    │                    │
       └────────────┬───────┴────────────────────┘
                    │
              ┌─────▼─────┐
              │ Cloud Sync │  (push/pull daemon or hook)
              │   Layer    │
              └─────┬──────┘
                    │
              ┌─────▼──────┐
              │ Cloud Store │  (S3 / Supabase / GCS)
              └─────┬──────┘
                    │
              ┌─────▼──────┐
              │  Shared     │  (optional)
              │  Birdview   │
              └─────────────┘
```

## Recommended Stacks by Team Size

### Small team (2-5 developers)
- **Store**: Supabase (free tier, real-time subscriptions built-in)
- **Sync**: PostToolUse hook that pushes changes via Supabase JS client
- **Dashboard**: Deploy birdview on Vercel/Netlify, connect to Supabase

### Medium team (5-20 developers)
- **Store**: S3 + DynamoDB for metadata
- **Sync**: Background daemon with conflict detection
- **Dashboard**: Internal deployment with SSO

### Large team (20+ developers)
- **Store**: Dedicated database (Postgres) with event sourcing
- **Sync**: Event-driven with message queue (SQS/Pub-Sub)
- **Dashboard**: Internal tool with role-based access

## Migration Steps

1. Deploy cloud store (follow provider-specific guide below)
2. Update `.omniscitus/cloud.yaml` with connection details
3. Set `enabled: true`
4. Each team member runs `/team-init` to verify cloud connectivity
5. Initial sync: push current `.omniscitus/` data to cloud
6. Verify all members see the same data

## Data Model (for custom backends)

### blueprints table/collection
- `project_id` (string) — identifies the repository
- `directory_key` (string) — e.g., "src", "_root", "plugins"
- `file_path` (string) — relative path
- `data` (JSON) — the blueprint entry
- `updated_by` (string) — git user identity
- `updated_at` (timestamp)

### history table/collection
- `project_id` (string)
- `unit_id` (string)
- `domain` (string)
- `content` (text) — markdown content
- `updated_by` (string)
- `updated_at` (timestamp)

## Conflict Resolution

The per-directory blueprint split minimizes conflicts by design:
- Developer A editing `src/` only touches `blueprints/src.yaml`
- Developer B editing `plugins/` only touches `blueprints/plugins.yaml`
- Same-directory conflicts use the configured `conflict_strategy`

For history units, append-only timeline entries rarely conflict.
If two developers wrap up the same unit simultaneously, merge both entries
chronologically.
```

## Rules

- Never deploy infrastructure automatically — only generate configs and guides
- cloud.yaml starts with `enabled: false` — user must explicitly enable
- Keep the guide provider-agnostic with specific examples
- Emphasize that git-based mode continues to work even after cloud is enabled (dual-write)
- The actual sync daemon/hook implementation is a separate engineering effort — this skill only scaffolds the config
